## Finding

### B-074 - OIDC discovery cache has no TTL

Lifecycle state: fixed
Severity: low
Protected principle: runtime readiness; operational boundedness; customer authority
Trust surface: hosted OIDC SSO provider discovery and callback processing
Repository evidence: `src/service/account-oidc.ts` stored provider discovery promises in a process-local `Map` without expiry.
Risk: IdP endpoint or JWKS metadata rotations could require an API process restart before hosted OIDC login used the refreshed provider configuration.

External anchors:
- OpenID Connect Discovery 1.0 defines the provider configuration document as the source for authorization endpoint, token endpoint, and `jwks_uri` metadata.
- NIST SP 800-115 anchors this as technical finding validation plus mitigation evidence.
- OWASP ASVS anchors this as application security verification evidence for session/federated-auth control behavior, not as certification.

Why applicable:
- Attestor's hosted OIDC login builds authorization requests and completes callbacks from discovered provider metadata.
- Stale discovery metadata can affect the customer authority boundary because the login flow may continue using a previous provider endpoint/JWKS configuration after IdP rotation.

Why not overclaimed:
- This remediation does not claim full OIDC conformance, IdP production readiness, or live provider verification.
- It does not audit all hosted account routes or replace OIDC library behavior; it only bounds Attestor's process-local discovery metadata cache.

Smallest safe fix:
- Add `ATTESTOR_HOSTED_OIDC_DISCOVERY_CACHE_TTL_SECONDS` with a default of 3600 seconds.
- Store discovery cache entries with an expiry timestamp and refresh provider metadata after expiry.
- Remove failed discovery attempts from the cache so transient provider errors do not become sticky until process restart.
- Clarify OIDC and SAML state-key fallback docs: production-like runtimes require dedicated keys; local/dev fallback remains non-production convenience only.

Negative/adversarial test:
- `tests/account-oidc-discovery-cache.test.ts` uses a local OIDC discovery endpoint whose `authorization_endpoint` changes after the first discovery.
- The test asserts the old endpoint is reused before TTL expiry and the new endpoint is used after TTL expiry, proving the original unbounded-stale-cache failure mode no longer persists indefinitely.

Positive/regression test or probe:
- The same test verifies discovery is not refetched before TTL expiry, preserving bounded caching rather than disabling cache behavior entirely.
- `npm run test:account-oidc-discovery-cache` is the targeted regression gate.

Remaining limitation:
- B-071 and B-073 remain accepted non-production limitations because production-like runtimes already reject admin-key fallback for SAML/OIDC state sealing.
- B-072 remains an accepted limitation: the SAML XML pre-guard is not a full XSD validator.

## Findings

### R1 B-003 - Monotone fusion threshold determinism

Status: validated and fixed repo-side.

Repository evidence before fix:

- `src/consequence-admission/relationship-aware-monotone-fusion.ts` reduced
  contribution arrays in caller-provided order.
- Threshold-relevant scores were returned directly from floating-point
  arithmetic.

Risk:

Different ordering of the same evidence set could produce different serialized
contribution order or threshold-adjacent score material. That weakens reviewer
evidence for deterministic control behavior.

### R1 C-001 - Hard-floor preservation declarative-only

Status: validated and fixed repo-side.

Repository evidence before fix:

- `RelationshipAwareMonotoneFusionResult.preservesHardFloor` was always
  returned as `true`.
- The function did not assert that a `hard_floor` input actually produced a
  `hard-floor-preserved` reason, hard-floor block pressure, and non-clear
  posture before returning the claim.

Risk:

A future regression could remove the behavior while leaving the claim literal
intact. That would turn proof material into self-attestation.

Why not overclaimed:

This remediation does not claim formal verification, calibrated risk scoring,
production readiness, compliance certification, or an external security audit.
It only proves repo-side deterministic ordering and hard-floor preservation
behavior for this fusion function.

## Smallest Safe Fix

- Sort opinions, relationships, modulators, and returned contributions by
  stable identifiers before scoring and returning output.
- Normalize threshold-relevant scores to fixed decimal precision.
- Runtime-assert that fused hazard does not fall below max raw input hazard.
- Runtime-assert that a hard-floor input yields `hard-floor-preserved`, retains
  hard-floor block pressure, and does not return `clear`.
- Add regression tests for low-score hard-floor preservation and shuffled input
  order stability.

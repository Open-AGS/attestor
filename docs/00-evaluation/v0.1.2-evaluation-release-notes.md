# Attestor v0.1.2-evaluation Release Notes

**Release type:** GitHub pre-release / attested evaluation baseline

**Tag:** `v0.1.2-evaluation`

**Package version:** `0.1.2-evaluation`

**Primary reviewer path:** [Attestor Evaluation Packet v0.1](v0.1-evaluation-packet.md)

This release keeps the v0.1 evaluation scope intact and adds a visible security/provenance baseline around the reviewer packet. It is meant to let an outside engineer see what is runnable, what is attested, and what is still outside the release boundary.

## What Changed Since v0.1.1-evaluation

- Added [Security Policy](../../SECURITY.md) with the supported evaluation status, vulnerability reporting guidance, scope boundaries, and CI token-permission posture.
- Added [Artifact attestation plan](../08-deployment/artifact-attestation-plan.md) describing the release-only provenance path and its non-claims.
- Added the `Release Provenance` GitHub Actions workflow for tagged evaluation releases and manual provenance runs.
- The provenance workflow renders `npm run proof:surface`, renders `npm run showcase:proof`, packages the review artifacts, uploads `evaluation-artifacts.tar.gz`, and publishes a GitHub artifact attestation with `actions/attest@v4`.
- Kept `Evaluation Smoke` and `Full Verify` on `permissions: contents: read`; elevated `attestations: write` and `id-token: write` remain isolated to the release provenance workflow.
- Added `npm run security:supply-chain-baseline` to check lockfile drift, registry resolution, integrity metadata, install-script allowlisting, and workflow permission regressions before dependency installation.
- Added `npm run test:security-baseline-docs` to protect the security policy, attestation plan, README entry, workflow permissions, and provenance non-claims.
- Split the longer live/ops verification path into explicit `verify:live-local`, `verify:ops`, and protected `verify:external-live` gates so external integrations are never implied by a secretless reviewer run.

## What To Run Locally

Start from a clean checkout at the release tag:

```bash
npm ci
npm run example:admission
npm run example:customer-gate
npm run proof:surface
```

For the portable proof-showcase path without external API credentials:

```bash
npm run showcase:proof
npm run verify:cert -- .attestor/showcase/latest/evidence/kit.json
```

`verify:cert` reports `PROOF_DEGRADED` and exits non-zero when the proof is local PostgreSQL-backed but upstream-model fixture-backed. That is expected for this evaluation release; use `npm run verify` as the green local release gate.

Before treating the release tag as reviewable:

```bash
npm run verify
```

## CI And Provenance Evidence

The release branch now has four workflow gates:

- `Evaluation Smoke`: fast push/PR gate for the reviewer path and proof-surface artifact.
- `Full Verify`: manual and weekly scheduled gate for `npm run verify`.
- `Security Scan`: high/critical npm audit, dependency review, CodeQL-adjacent security baseline coverage, and the supply-chain baseline guard.
- `Release Provenance`: tag/manual workflow that uploads and attests `evaluation-artifacts.tar.gz`.

After downloading `evaluation-artifacts.tar.gz` from a `Release Provenance` run, reviewers can verify provenance with:

```bash
gh attestation verify evaluation-artifacts.tar.gz -R 0xlamarr-labs/attestor --signer-workflow 0xlamarr-labs/attestor/.github/workflows/release-provenance.yml
```

The provenance workflow is a trust signal for the evaluation artifact bundle. It does not replace production deployment rehearsal, security audit, customer-operated validation, or full supply-chain provenance for external environments.

- A short local admission path with one admitted consequence and one blocked fail-closed consequence.
- A customer-side gate that proceeds only when Attestor returns an allowed decision.
- Local proof-surface rendering under `.attestor/proof-surface/latest/`.
- A CI-backed proof-surface artifact from the `Evaluation Smoke` workflow.
- A release evaluation artifact bundle with GitHub artifact attestation from the `Release Provenance` workflow.

- A longer live/ops verification path where local live and ops render/probe checks are separated from opt-in external live integrations.

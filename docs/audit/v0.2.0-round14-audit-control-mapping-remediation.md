## Finding

### B-053 - Meta-finding anchors need applicability and no-overclaim rationale

Status: fixed for repository-side lifecycle records.

Repository evidence:

- `docs/audit/finding-lifecycle-and-evidence-ledger.md` now requires external
  anchors, applicability rationale, and no-overclaim rationale as closure
  evidence.
- `docs/audit/ledger-template.md` exposes the same fields in the compact
  finding template.
- `scripts/check/check-audit-finding-evidence.mjs` rejects closed lifecycle records
  with empty control-mapping fields.
- `tests/audit-finding-lifecycle.test.ts` verifies the schema, example record,
  package script, and evidence checker.

External anchors:

- NIST SP 800-115: technical assessment findings should produce evidence and
  mitigation reporting, not unsupported closure assertions.
- NIST CSF 2.0 Govern function: policy, oversight, and risk-management context
  need documented governance framing.
- OWASP ASVS: verification evidence should map to testable security
  requirements.
- OWASP DSOVS: security issue tracking and security test management should be
  explicit parts of DevSecOps verification.

Why applicable:

The finding concerns audit-governance evidence. These sources apply because the
change controls how Attestor records, maps, verifies, and closes security and
assurance findings.

Why not overclaimed:

The mapping is engineering evidence for repository-side remediation discipline.
It is not an external audit, SOC 2 report, ISO certification, OWASP
certification, NIST approval, production readiness claim, or compliance claim.

## Smallest Safe Fix

Require three explicit fields in closed lifecycle records:

- `External anchors:`
- `Why applicable:`
- `Why not overclaimed:`

Keep the historical tracker at minimum evidence posture instead of forcing old
rows into false full-control mappings.

# Attestor v0.3.0-evaluation Release Notes

**Release type:** GitHub pre-release / multi-path evaluation baseline

**Tag:** `v0.3.0-evaluation`

**Package version:** `0.3.0-evaluation`

**Primary reviewer path:** [Try Attestor first](../01-overview/try-attestor-first.md)

**Integration path:** [How to integrate Attestor](../01-overview/how-to-integrate-attestor.md)

This release keeps Attestor in evaluation scope, but moves the review surface
beyond the first refund Golden Path. The repository now presents a clearer
local run, shadow pilot path, runtime-signal path, and customer-gate integration
path while preserving the boundary that live customer enforcement and
production readiness require separate proof.

Versioning note: `v0.3.0-evaluation` follows the repository's existing
`<semver>-evaluation` convention and uses `evaluation` as a
SemVer-compatible pre-release identifier under [Semantic Versioning
2.0.0](https://semver.org/) and npm's
[package versioning guidance](https://docs.npmjs.com/about-semantic-versioning).
It is not `beta`, not `rc`, not GA, and not a production release.

## What Changed Since v0.2.0-evaluation

- Added multi-path local evaluation across Money Movement, Data Movement,
  Authority Change, External Communication, Operational Execution, and
  Programmable Money.
- Added runtime-signal handling for turning OpenAPI, MCP, and OpenTelemetry
  metadata into digest-only signals, consequence candidates, and
  gate-placement/readiness plans.
- Added the Data Movement consequence-engine proof track for the repository-side
  path from admission through release token, introspection, replay consumption,
  release-enforcement PEP, sandbox export gate, and proof packet.
- Hardened generic admission behavior around guard wiring, tenant and shadow
  boundaries, idempotency, state isolation, and fail-closed invariants.
- Added customer middleware and integration examples for placing the
  customer-owned gate before real service calls.
- Added the review surface contract, export, preview, and hosted route path for
  human-readable decision review.
- Cleaned up first-reader navigation with a shorter README, shadow pilot entry,
  integration guide, repository navigator, and developer entry path.
- Refreshed dependency and supply-chain evidence while keeping the release in
  evaluation scope.

## CI And Provenance Evidence

The release keeps the reviewer and provenance gates from the evaluation
baseline. The release security boundary remains governed by the
[Security Policy](../../SECURITY.md):

- `Evaluation Smoke`: fast push/PR gate for the reviewer path and proof-surface artifact.
- `Full Verify`: manual and weekly scheduled gate for `npm run verify`.
- `Security Scan`: high/critical npm audit, dependency review, CodeQL-adjacent security baseline coverage, and the supply-chain baseline guard.
- `Release Provenance`: tag/manual workflow that uploads and attests `evaluation-artifacts.tar.gz` through `actions/attest@v4`.

After downloading `evaluation-artifacts.tar.gz` from a `Release Provenance` run,
reviewers can verify provenance with:

```bash
gh attestation verify evaluation-artifacts.tar.gz -R AI-gateway-systems/attestor --signer-workflow AI-gateway-systems/attestor/.github/workflows/release-provenance.yml
```

The provenance workflow is a trust signal for the evaluation artifact bundle.
It does not replace production deployment rehearsal, security audit,
customer-operated validation, external KMS/HSM proof, or full supply-chain
provenance for external environments.

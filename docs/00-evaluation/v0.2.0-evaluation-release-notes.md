# Attestor v0.2.0-evaluation Release Notes

**Release type:** GitHub pre-release / Golden Path evaluation baseline

**Tag:** `v0.2.0-evaluation`

**Package version:** `0.2.0-evaluation`

**Primary reviewer path:** [Golden Path: Refund](../02-architecture/golden-refund-shadow-pilot.md)

**Secondary reviewer path:** [Attestor Evaluation Packet v0.1](v0.1-evaluation-packet.md)

This release keeps Attestor in evaluation scope, but changes the public review
posture from "basic admission and proof-surface baseline" to a concrete
end-to-end Golden Path. The first path a reviewer should run is now a synthetic,
shadow-only refund scenario that exercises the Attestor consequence engine
without executing a refund or activating policy.

Versioning note: `v0.2.0-evaluation` follows the repository's existing
`<semver>-evaluation` convention and uses `evaluation` as a
SemVer-compatible pre-release identifier under [Semantic Versioning
2.0.0](https://semver.org/) and npm's
[package versioning guidance](https://docs.npmjs.com/about-semantic-versioning).
It is not `beta`, not `rc`, not GA, and not a production release.

## What Changed Since v0.1.2-evaluation

- Added [Golden Path: Refund](../02-architecture/golden-refund-shadow-pilot.md) as the first end-to-end repo path.
- Added `npm run demo:golden-refund`, with Markdown-first output and JSON as a secondary machine format.
- Added digest-only refund shadow fixtures for normal, missing-evidence, stale-evidence, repeated-refund, and approval-required paths.
- Added the Golden Path refund projection into Policy Foundry summary material with named evidence, authority, relationship, and approval gaps.
- Added refund runtime assurance smoke over the R02-R07 shadow runtime chain.
- Added a pilot readiness probe for the Golden Path that can return only `ready-for-shadow-pilot` or `not-ready`; `ready-for-scoped-pilot` remains outside this path.
- Completed the Shadow-to-Policy master plan repository-side: 26/26 steps complete.
- Completed Runtime Activation R01-R08 repository-side, including the end-to-end fixture replay smoke.
- Completed the consequence runtime assurance contract stack, wiring stack, and intelligence/assurance-case contract tracks in their stated repository scopes.
- Completed the production-shared authority-plane track repository-side with embedded PostgreSQL shared-authority behavior, while keeping external customer-operated production readiness out of scope.
- Hardened the README first impression around Golden Path, proof vocabulary, customer-side PEP dependency, and production-boundary non-claims.
- Preserved the v0.1.2 attested evaluation baseline: `Evaluation Smoke`, `Full Verify`, `Security Scan`, and `Release Provenance` remain the release evidence path.

## What To Run Locally

Start from a clean checkout at the release tag:

```bash
npm ci
npm run demo:golden-refund
npm run demo:golden-refund -- --json
```

Then inspect the lower-level primitives:

```bash
npm run example:admission
npm run example:customer-gate
npm run proof:surface
```

For the portable proof-showcase path without external API credentials:

```bash
npm run showcase:proof
npm run verify:cert -- .attestor/showcase/latest/evidence/kit.json
```

`verify:cert` reports `PROOF_DEGRADED` and exits non-zero when the proof is
local PostgreSQL-backed but upstream-model fixture-backed. That is expected for
this evaluation release; use `npm run verify` as the green local release gate.

Before treating the release tag as reviewable:

```bash
npm run verify
```

## CI And Provenance Evidence

The release branch keeps the reviewer and provenance gates from the attested
evaluation baseline. The release security boundary remains governed by the
[Security Policy](../../SECURITY.md):

- `Evaluation Smoke`: fast push/PR gate for the reviewer path and proof-surface artifact.
- `Full Verify`: manual and weekly scheduled gate for `npm run verify`.
- `Security Scan`: high/critical npm audit, dependency review, CodeQL-adjacent security baseline coverage, and the supply-chain baseline guard.
- `Release Provenance`: tag/manual workflow that uploads and attests `evaluation-artifacts.tar.gz` through `actions/attest@v4`.

After downloading `evaluation-artifacts.tar.gz` from a `Release Provenance` run,
reviewers can verify provenance with:

```bash
gh attestation verify evaluation-artifacts.tar.gz -R AI-gateway-systems/attestor --signer-workflow AI-gateway-systems/attestor/.github/workflows/release-provenance.yml
```

The provenance workflow is a trust signal for the evaluation artifact bundle. It
does not replace production deployment rehearsal, security audit,
customer-operated validation, external KMS/HSM proof, or full supply-chain
provenance for external environments.

## What This Evaluation Release Demonstrates

- A concrete Golden Path: Refund story for the Attestor consequence engine.
- A synthetic support refund action surface from a bounded OpenAPI fixture.
- Digest-only canonical shadow fixtures.
- Runtime assurance smoke over refund scenarios.
- Policy Foundry summary material with named gaps.
- A pilot readiness packet limited to `ready-for-shadow-pilot` or `not-ready`.
- A bounded decision vocabulary: `admit`, `narrow`, `review`, `block`.
- A short local admission path with one admitted consequence and one blocked fail-closed consequence.
- A customer-side gate that proceeds only when Attestor returns an allowed decision.
- Local proof-surface rendering under `.attestor/proof-surface/latest/`.
- A CI-backed proof-surface artifact from the `Evaluation Smoke` workflow.
- A release evaluation artifact bundle with GitHub artifact attestation from the `Release Provenance` workflow.
- Independent kit inspection through `npm run verify:cert -- .attestor/showcase/latest/evidence/kit.json`, with expected `PROOF_DEGRADED` posture unless a live upstream model is configured.
- A full local verification gate through `npm run verify`.
- A longer live/ops verification path where local live and ops render/probe checks are separated from opt-in external live integrations.

# Attestor Evaluation Packet v0.1

This packet is a small, runnable proof of selected claims Attestor can demonstrate today.

It is meant for an outside engineer, buyer, or technical reviewer who wants to see the system work before reading the deeper architecture docs.

For the release boundary, verification gate, and known limitations around the current evaluation tag, see [v0.3.0-evaluation release notes](v0.3.0-evaluation-release-notes.md).

## Evaluation Scenario

**AI output -> structured financial record release**

A customer system wants to let an AI-assisted workflow affect a real financial record or controlled downstream action.

Attestor sits before that consequence. The customer system proposes the action to Attestor, Attestor evaluates policy, authority, evidence, and release constraints, then returns a bounded decision:

- `admit`: proceed as requested
- `narrow`: proceed only with a safer bounded version
- `review`: require human or external review
- `block`: reject fail-closed

The downstream system proceeds only when the decision allows it.

## Reviewer Paths

Start from a clean checkout:

```bash
npm ci
```

### If You Have 15 Minutes

Run the shortest local admission proof:

```bash
npm run example:admission
```

Expected markers:

- `Scenario: Allowed finance consequence`
- `canonical: admit`
- `Proof refs:` followed by certificate / verification references
- `Downstream result:` with `PROCEED`
- `Scenario: Blocked finance consequence`
- `canonical: block`
- `Proof refs:` followed by `none`
- `Downstream result:` with `HOLD`

Then run the customer-side gate:

```bash
npm run example:customer-gate
```

Expected markers:

- `Customer gate: PROCEED` for the allowed path
- `Customer gate: HOLD` for the blocked path
- the customer gate writes/sends only when Attestor returns an allowed decision
- downstream code does not run just because a model, agent, or workflow proposed an action

Render the local proof surface:

```bash
npm run proof:surface
```

This writes `.attestor/proof-surface/latest/` with a manifest, machine-readable bundle, markdown summary, and unified proof outputs for runnable scenarios.

### If You Have 45 Minutes

Generate a concrete proof showcase packet and inspect the resulting kit outside the main runtime:

```bash
npm run showcase:proof
npm run verify:cert -- .attestor/showcase/latest/evidence/kit.json
```

Expected markers:

- `Attestor proof showcase packet created.`
- `.attestor/showcase/latest/README.md`
- `.attestor/showcase/latest/index.html`
- `.attestor/showcase/latest/packet.json`
- `Overall: PROOF_DEGRADED` from `verify:cert` when no live upstream model is configured

`verify:cert` exits non-zero for `PROOF_DEGRADED`; that is expected for the local PostgreSQL-backed proof when the upstream model is fixture-backed. The full local release gate is still `npm run verify`.

Optional live-upstream variant:

```bash
npm run showcase:proof:hybrid
```

That variant requires `OPENAI_API_KEY` and may create source run artifacts under `.attestor-financial/runs/`. The default reviewer packet is copied to `.attestor/showcase/latest/`.

## Artifact Map

| Command | Primary output | What to inspect | Why it matters |
|---|---|---|---|
| `npm run example:admission` | terminal output | `canonical: admit`, `canonical: block`, `PROCEED`, `HOLD`, proof refs | Shows the smallest admission path and fail-closed behavior. |
| `npm run example:customer-gate` | terminal output | `Customer gate: PROCEED`, `Customer gate: HOLD`, proof refs | Shows the downstream gate refusing unsafe execution. |

| `npm run showcase:proof:hybrid` | `.attestor/showcase/latest/` and `.attestor-financial/runs/` | live-upstream proof packet when `OPENAI_API_KEY` is configured | Optional live-model variant, not the default reviewer path. |
| `npm run verify` | terminal output | full local gate completion | Shows the repository's local verification gate passes. |

## Failure Modes To Check

These commands keep the evaluation honest. They exercise the packet beyond the happy path.

```bash
npm run test:first-useful-admission-demo
npm run test:consequence-admission-customer-gate
npm run test:proof-surface-readiness
```

What to look for:

- blocked actions fail closed
- missing required proof cannot quietly pass through the customer gate
- proof-surface output is local static evidence, not a hosted console claim

## Production-Shared Truth Gates

The production-shared authority path is not declared ready from README text or configuration labels alone. It requires explicit runtime posture and shared authority store wiring.

Use these checks when reviewing production-shared readiness:

```bash
npm run test:production-shared-request-path-cutover
npm run test:production-shared-multi-instance-recovery
npm run test:production-readiness-packet
```

Truths this packet relies on:

- `ATTESTOR_RUNTIME_PROFILE=production-shared` is explicit
- `ATTESTOR_RELEASE_AUTHORITY_PG_URL` is required for production-shared authority stores
- request-path shared-store readiness must be backed by the async shared authority store contract
- readiness must stay blocked when required shared stores are missing, unreachable, mismatched, or explicitly blocked

These tests use local/embedded harnesses where appropriate. They are not a claim that a customer-operated external production deployment has already been completed.

## Full Local Gate

For a full local repository gate:

```bash
npm run verify
```

This is heavier than the packet-specific checks. Use it before tagging a release candidate or sending the packet to a serious reviewer.

## What This Proves

- Attestor can gate AI-assisted consequences before downstream write, send, file, or execute actions.
- The customer-facing decision vocabulary is bounded: `admit`, `narrow`, `review`, `block`.
- Blocked actions fail closed.
- Allowed actions can carry proof references and durable evidence.
- A customer-side gate can refuse to run downstream actions without an allowed Attestor decision.
- Production-shared readiness is guarded by explicit runtime profile and shared authority store requirements.
- The finance path is the deepest proven evaluation scenario today.

## What This Does Not Claim

- Not a hosted public SaaS launch.
- Not a completed customer-operated deployment.
- Not a public hosted crypto HTTP route.
- Not automatic pack detection.
- Not a replacement for a customer's model, agent runtime, wallet, custody platform, orchestration layer, or downstream system.
- Not market validation or paid customer adoption.
- Not proof that every future domain pack is production ready.

## Review Checklist

Use this packet to answer five questions:

- Can I run the first admission demo locally?
- Can I see an allowed action and a blocked fail-closed action?
- Can I see where proof references and evidence outputs live?
- Can I see that downstream execution is gated by Attestor's decision?
- Can I see that production-shared readiness is guarded instead of overclaimed?

## Finding

### B-057 - `maxTtlSeconds ?? 300` permissive default

Lifecycle state: fixed
Severity: low
Protected principle: fail-closed boundary; customer authority; operational boundedness
Trust surface: release-token exchange policy boundary
Repository evidence: `requestedTtlSeconds()` used `input.policy.maxTtlSeconds ?? 300`, allowing an exchange to issue a downstream token when the policy omitted an explicit maximum TTL.
Risk: a valid audience/scope exchange could receive a five-minute downstream release token from an implicit default rather than an explicit policy decision.

External anchors:
- RFC 8693 section 2.2.1 defines successful token exchange response lifetime reporting through `expires_in`, and section 5 notes limited token lifetime as a mitigation for delegated-rights abuse.
- NIST SP 800-115 anchors this as mitigation tracking and verification evidence for a technical security finding.
- OWASP ASVS anchors this as application security verification evidence for explicit control behavior, not as certification.

Why applicable:
- Attestor's token exchange is an OAuth-style release-token narrowing boundary. The issued token's lifetime is part of the authorization policy, so an implicit default weakens the policy evidence trail.
- The failure mode is not general OAuth non-compliance; it is an Attestor-specific no-claim and fail-closed boundary issue.

Why not overclaimed:
- This remediation does not claim RFC 8693 certification, production readiness, or complete release-enforcement-plane audit coverage.
- It does not derive risk-class TTLs or audit all token-exchange callers; it only requires the exchange policy to carry an explicit positive maximum TTL.

Smallest safe fix:
- Make `ReleaseTokenExchangePolicy.maxTtlSeconds` required at the TypeScript contract level.
- Deny exchange with `ttl-policy-required` when runtime input omits or invalidates the explicit maximum TTL.
- Preserve existing narrowing: child token TTL remains the minimum of subject token remaining lifetime, policy max TTL, and requested TTL.

Negative/adversarial test:
- `tests/release-enforcement-plane-token-exchange.test.ts` now submits an otherwise valid audience/scope exchange with no `maxTtlSeconds` and asserts denial with `ttl-policy-required` and no issued token.

Positive/regression test or probe:
- Existing successful exchange tests now provide explicit `maxTtlSeconds`, preserving audience, scope, actor-chain, subject-lifetime, and high-risk online-verifier behavior.

Remaining limitation:
- This slice does not close Round 18 targets: `offline-verifier.ts`, `http-message-signatures.ts`, `workload-binding.ts`, or `async-envelope.ts`.

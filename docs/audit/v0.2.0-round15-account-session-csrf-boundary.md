# v0.2.0 Round 15 Account Session CSRF Boundary

Status: accepted limitation with repo-side guard documentation

## Scope

Audit source: Round 14 account-route CSRF verification follow-up.

Affected trust surface:

- hosted account session cookie mutations
- account-plane API routes guarded by `requireAccountSession`
- deployment CORS boundary for cookie-authenticated account mutations

Protected principles:

- customer authority
- fail-closed boundary
- no overclaim
- runtime readiness

## Finding

### R14-CSRF-001 - CSRF check validates custom-header presence, not a session-bound token

Status: accepted limitation for the current repository scope.

Repository evidence:

- `src/service/request-context.ts` requires `x-attestor-csrf` for unsafe
  cookie-authenticated account mutations.
- `src/service/request-context.ts` does not compare the header value against a
  session-bound CSRF secret.
- `src/service/http-production-edge-contract.ts` does not install permissive
  `Access-Control-Allow-Origin` headers.
- `src/service/api-server.ts` does not install Hono CORS middleware.
- `docs/08-deployment/deployment.md` now states the boundary is
  header-presence CSRF, not a synchronizer-token or double-submit-token claim.

External anchors:

- OWASP CSRF Prevention Cheat Sheet: custom request headers are an accepted
  AJAX/API CSRF defense pattern when CORS is correctly restricted.
- OWASP ASVS: anti-CSRF controls should be explicit and verifiable for
  state-changing requests.
- MDN CORS documentation: non-simple requests with custom headers require a
  preflight and server opt-in.

Why applicable:

The finding concerns browser-mediated cross-site account mutations. The
applicable control is whether a malicious cross-origin page can cause an
authenticated cookie-backed account mutation. The current repository defense is
an explicit custom-header requirement plus no repo-side permissive CORS.

Why not overclaimed:

This closure does not claim a synchronizer token, double-submit token, complete
browser security posture, customer deployment CORS correctness, WAF coverage,
or XSS resistance. A deployment that adds wildcard, broad-regex, or
customer-controlled credentialed CORS would invalidate this boundary.

## Decision

Accepted limitation, not a code vulnerability as stated.

The report's observation is accurate: the header value is not session-bound.
However, OWASP documents custom-header presence as a valid AJAX/API CSRF
pattern when CORS is restricted. The smallest safe repository fix is to make the
claim boundary explicit and test that Attestor does not present this as
double-submit-token protection.
